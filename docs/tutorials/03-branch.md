# 🌿 فصل ۳: شاخه‌ها (Branch) در گیت

<div dir="rtl">

**🎯 تو این فصل چی یاد می‌گیریم؟**

- شاخه یا Branch چیست و چرا استفاده می‌شود
- ساختن شاخه جدید
- جابه‌جایی بین شاخه‌ها
- ادغام شاخه‌ها با `merge`
- حل تعارض یا Merge Conflict
- حذف شاخه‌هایی که کارشان تمام شده

## شاخه یعنی چی؟

شاخه در گیت یعنی یک مسیر جدا برای کار کردن روی پروژه. شاخه اصلی بیشتر پروژه‌ها معمولاً `main` است. وقتی می‌خواهی یک ویژگی جدید بسازی یا یک باگ را درست کنی، بهتر است روی یک شاخه جدا کار کنی تا نسخه اصلی پروژه تمیز و قابل استفاده بماند.

فرض کن داری روی سایت شخصی‌ات کار می‌کنی:

- شاخه `main`: نسخه سالم و قابل نمایش سایت
- شاخه `feature-contact-form`: ساخت فرم تماس
- شاخه `fix-navbar`: رفع مشکل منوی بالای سایت

تا وقتی کارت کامل نشده، تغییرات را در شاخه جدا نگه می‌داری. وقتی مطمئن شدی همه چیز درست است، آن را به `main` برمی‌گردانی.

## دیدن شاخه‌ها

</div>

```bash
git branch
```

<div dir="rtl">

خروجی ممکن است شبیه این باشد:

</div>

```bash
* main
  feature-navbar
```

<div dir="rtl">

علامت `*` یعنی الان روی همان شاخه هستی.

## ساختن شاخه جدید

</div>

```bash
git branch feature-navbar
```

<div dir="rtl">

این دستور فقط شاخه را می‌سازد، اما تو را وارد آن شاخه نمی‌کند. برای رفتن به شاخه جدید:

</div>

```bash
git switch feature-navbar
```

<div dir="rtl">

راه سریع‌تر:

</div>

```bash
git switch -c feature-navbar
```

<div dir="rtl">

این دستور هم شاخه را می‌سازد، هم همان لحظه واردش می‌شود.

## کار کردن روی شاخه

بعد از ورود به شاخه، مثل همیشه فایل‌ها را تغییر می‌دهی و کامیت می‌کنی:

</div>

```bash
echo "<nav>منوی سایت</nav>" > navbar.html
git add navbar.html
git commit -m "اضافه کردن navbar"
```

<div dir="rtl">

این کامیت فقط روی شاخه `feature-navbar` ثبت می‌شود و تا وقتی merge نکنی، وارد `main` نمی‌شود.

## جابه‌جایی بین شاخه‌ها

</div>

```bash
git switch main
git switch feature-navbar
```

<div dir="rtl">

قبل از جابه‌جایی، بهتر است وضعیت پروژه را ببینی:

</div>

```bash
git status
```

<div dir="rtl">

اگر تغییرات ذخیره‌نشده داشته باشی، گیت ممکن است اجازه جابه‌جایی ندهد. معمولاً یا باید کامیت کنی، یا تغییرات را با `stash` موقتاً کنار بگذاری.

## ادغام شاخه‌ها با merge

وقتی کار شاخه کامل شد، باید آن را به شاخه اصلی اضافه کنیم:

</div>

```bash
git switch main
git merge feature-navbar
```

<div dir="rtl">

اگر همه چیز ساده باشد، گیت خودش تغییرات را ادغام می‌کند.

## حذف شاخه تمام‌شده

بعد از merge، شاخه کاری معمولاً دیگر لازم نیست:

</div>

```bash
git branch -d feature-navbar
```

<div dir="rtl">

اگر شاخه هنوز merge نشده باشد، گیت جلوی حذف امن را می‌گیرد. حذف اجباری این است، ولی فقط وقتی استفاده کن که مطمئنی:

</div>

```bash
git branch -D feature-navbar
```

<div dir="rtl">

## تعارض یا Merge Conflict

تعارض وقتی پیش می‌آید که دو شاخه یک بخش از یک فایل را به شکل متفاوت تغییر داده باشند و گیت نتواند خودش تصمیم بگیرد کدام درست است.

مثلاً داخل فایل `about.txt` ممکن است چنین چیزی ببینی:

</div>

```text
<<<<<<< HEAD
این متن از شاخه main آمده است
=======
این متن از شاخه feature-about آمده است
>>>>>>> feature-about
```

<div dir="rtl">

برای حل تعارض:

1. فایل را باز کن
2. متن درست را نگه دار
3. علامت‌های `<<<<<<<` و `=======` و `>>>>>>>` را حذف کن
4. فایل را ذخیره کن
5. فایل را add و commit کن

</div>

```bash
git add about.txt
git commit -m "حل تعارض about"
```

<div dir="rtl">

## دستورهای مهم این فصل

</div>

```bash
git branch
git switch -c feature-name
git switch main
git merge feature-name
git branch -d feature-name
```

<div dir="rtl">

## جمع‌بندی

شاخه‌ها به تو اجازه می‌دهند بدون خراب کردن نسخه اصلی پروژه، آزادانه تجربه کنی. قانون ساده این فصل این است: برای هر ویژگی یا باگ، یک شاخه جدا بساز؛ وقتی آماده شد، آن را به `main` ادغام کن.

</div>
